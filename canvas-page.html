📖 Student Directory: Hash Table Implementation

🎯 Overview

In this assignment you will build a student directory using two hash table implementations from scratch, practicing every core concept from Module 5. Student names are the keys and student IDs are the values — the same scenario you worked with in CT10 and CT11.

  • Hash functions — converting string keys to array indices using a custom hash algorithm
  • Modulo mapping — mapping hash values into a fixed-size bucket array
  • Separate chaining (open hashing) — resolving collisions with linked lists at each bucket
  • Linear probing (closed hashing) — resolving collisions by scanning forward in the array
  • Tombstone deletion — marking slots as deleted so probing sequences aren't broken
  • Load factor — tracking how full the table is (size / capacity)
  • Resizing & rehashing — growing the table and reinserting all entries when the load factor is too high

📖 Learning Objectives

By completing this assignment, you will practice:

  • Custom hash functions — implementing a hash algorithm that converts strings to integer indices
  • Separate chaining — building and traversing linked-list chains at each bucket
  • Linear probing — finding open slots by scanning forward with wraparound
  • Tombstone deletion — marking deleted slots so probe sequences aren't broken
  • Load factor & resizing — detecting when to grow the table and rehashing all entries
  • Memory management — allocating and freeing dynamic arrays and linked-list nodes

💡 Key Concepts

Chaining vs. Probing
Chaining stores collisions in linked lists hanging off each bucket — the array never fills up, but chains can grow long. Probing stores everything directly in the array by scanning forward for an open slot — more cache-friendly, but requires tombstones for deletion.
Load Factor & Resizing
Load factor = size / capacity. When it exceeds the threshold (typically 0.75), allocate a larger array with a prime capacity and rehash every existing entry — hash values change because the capacity changed.
Common Pitfall: Tombstones
When probing, you must skip past DELETED (tombstone) slots during search() — stopping early would miss entries inserted before the deletion. During insert(), you can reuse tombstone slots. During resize(), skip tombstones entirely — only rehash OCCUPIED slots.

🔧 Troubleshooting

  1. Make sure you have Desktop development with C++ workload installed in Visual Studio
  2. Go to Project → CMake Settings and ensure the configuration is correct
  3. Try Project → Delete Cache and Reconfigure
  4. Verify that CMakeLists.txt is in the root of your cloned repository

Tests fail to build

  • Google Test is fetched automatically via CMake's FetchContent — make sure you have an internet connection during the first build
  • Do not modify header files or test files

Running Tests

  1. In Visual Studio, select run_tests.exe as the startup item from the dropdown
  2. Press Ctrl+F5 to run without debugging
  3. Check the output for passed/failed test results
